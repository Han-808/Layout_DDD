Questions to be settled
=======================

1. Exact evaluator thresholds
   - near / far distance
   - against_wall tolerance
   - facing angle tolerance
   - support overlap ratio
   - floor contact tolerance

2. Spatial relation definitions
   - Whether relations are center-based, bbox-edge-based, or category-specific.
   - Whether left/right/front/behind should be global-room-frame or object-facing-frame.
   - How strict soft constraints should be compared with hard constraints.

3. Physical validity scope
   - Whether collision checks should stay yaw-aware bbox intersection only.
   - Whether to add category-specific clearance / reachability rules.
   - Whether support rules should be category-aware.

4. Repair benchmark protocol
   - Whether repair uses exactly the same model by default.
   - How many repair iterations should be reported: success@1, success@2, etc.
   - Whether valid objects should be locked strictly or only instruction-level preserved.

5. Model protocol
   - Which real target model adapters to implement first:
     OpenAI-compatible, Anthropic-compatible, Gemini-compatible, local vLLM.
   - Whether prompts should include few-shot examples.
   - Whether model outputs should be accepted only as raw JSON or allow JSON extraction from text.

6. Dataset design
   - How many benchmark cases per level.
   - Whether hidden_annotations should remain unused in MVP.
   - How to define expected object coverage and expected spatial relations.

7. Visualization scope
   - Static Three.js viewer is visualization-only.
   - Decide later whether to build a fuller frontend.
   - Decide later whether to add Blender renders or three-view VLM-as-judge as auxiliary outputs.

Current workflow
================

Benchmark workflow
------------------

1. Load benchmark instance
   - Input file: bm_instance.json
   - Contains task_id, input_mode, scene_prompt, optional room, optional required_objects,
     spatial_constraints, optional hidden_annotations.

2. Generate initial layout
   - GenerateLayoutNode calls the target model.
   - In the current demo, MockModel intentionally creates an inconsistent layout.
   - Output: initial_layout_iter_0.json

3. Deterministic evaluation
   - EvaluateLayoutNode runs Python evaluator.
   - Checks schema validity, physical validity, and spatial relation validity.
   - Output: evaluation_report_iter_0.json

4. Deterministic feedback
   - BuildFeedbackNode / feedback_builder converts evaluator failures into feedback.
   - No LLM is used to write feedback.json.
   - Output: feedback_iter_0.json

5. Repair once
   - RepairLayoutNode calls the same target model by default.
   - In the current demo, MockModel repairs the inconsistent layout.
   - Output: repaired_layout_iter_1.json

6. Re-evaluate repaired layout
   - EvaluateLayoutNode runs again on the repaired layout.
   - Output: evaluation_report_iter_1.json

7. Compute metrics
   - Metrics include schema_validity, physical_validity, spatial_relation_validity,
     overall_validity, success_at_k, and error reduction.
   - Output: per_case_result.json

8. Export viewer scene
   - threejs_adapter exports viewer_scene.json.
   - The viewer scene includes room, objects, relations, violations, history,
     workflow artifacts, deterministic feedback, and smoke-test metrics.


Interactive demo workflow
-------------------------

Run:

  py scripts/run_demo.py --out outputs/demo
  cd outputs/demo
  py -m http.server 8000

Open:

  http://localhost:8000/?v=topbar

Expected interaction:

1. Click "Before Repair"
   - Shows iteration 0.
   - Layout is intentionally invalid.
   - Invalid objects are highlighted.
   - Violations are visible.

2. Click "Build deterministic feedback"
   - Shows feedback_iter_0.json in the Artifact JSON drawer.
   - Shows repair_targets, locked_objects, and violation messages.

3. Click "After Repair"
   - Shows iteration 1.
   - Layout is valid after one repair.
   - Violations should be empty.

4. Click "Replay Workflow"
   - Automatically steps through:
     input -> initial layout -> initial evaluation -> feedback -> repair ->
     repaired evaluation -> metrics -> viewer scene.

5. Click any workflow step
   - The 3D view switches to the relevant iteration when applicable.
   - The Artifact JSON drawer shows the corresponding JSON artifact.

Important constraint:
The Three.js viewer is visualization-only. It does not compute benchmark metrics.
